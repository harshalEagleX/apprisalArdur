STEP 1 — CONFIDENCE ROUTING AFTER EXTRACTION
85%+
Auto QC decision
Rule evaluates and stores result as PASS or FAIL automatically. No reviewer needed.
60–85%
Standard reviewer queue
Reviewer sees extracted value with source page highlight. One-click confirm or correct.
<60%
Ollama assist + reviewer
Local Ollama suggests a value for the uncertain field. Its output is treated as untrusted context. Reviewer confirms or overrides.
Decision boundary: Python extracts facts and confidence only. Ollama may propose a field value for low-confidence cases. Java rules decide PASS, FAIL, VERIFY, AUTO_FAIL, or SKIPPED. Reviewer corrections are the training signal.
STEP 2 — REVIEWER CORRECTS, CORRECTION IS STORED
correction_log {
  qc_result_id, field_catalog_id,
  extracted_value, corrected_value,
  confidence_score, extraction_method,
  source_page, source_bbox,
  reviewer_id, reviewed_at,
  pattern_candidate_id
}
STEP 3 — PATTERN MINER RUNS NIGHTLY
Groups corrections by field_key + extraction_method + source_context. Detects patterns like "subject.address is wrong when page type is scanned header". When N ≥ 5 consistent corrections match, the pattern is flagged for admin review as a Java rule, Python regex, or extraction heuristic.
pattern_candidate {
  field_key, extraction_method,
  source_context_hint,
  sample_correction_count,
  proposed_fix,
  status,
  promoted_rule_code
}
STEP 4 — PATTERN PROMOTED TO RULE
Admin approves the promoted regex, heuristic, or deterministic Java rule. Java rule engine picks it up on next deploy. Future extractions of that pattern get higher confidence and fewer reviewer interruptions over time.
No external AI dependency. Python uses regex + PyMuPDF + Tesseract on CPU. Ollama is local, optional, and advisory. Pattern mining is SQL aggregation. Rules remain deterministic and auditable in Java.
